# State and configuration inventory

This is the operator contract for durable EduFlow state, configuration, assets, and credentials. In the table, `$STATE` means the resolved `EDUFLOW_STATE_DIR`.

## Path authority and known drift

The Python authority is `src/eduflow/runtime/paths.py`: `EDUFLOW_STATE_DIR` wins and bare Python falls back to `~/.eduflow`. The repository wrapper and production launch convention set `EDUFLOW_STATE_DIR=$PROJECT_ROOT/.eduflow-team-state`. Therefore `.eduflow-team-state` and `~/.eduflow` are not aliases. Running `python -m eduflow.cli` without the wrapper/environment can silently create or read a second state tree. Recovery and migration must identify the live process environment before copying either tree.

Unified configuration resolves through `EDUFLOW_CONFIG_FILE`, otherwise `./eduflow.toml`. During migration, `src/eduflow/runtime/config.py` still supports legacy/fallback `EDUFLOW_TEAM_FILE` or `./team.json` and `EDUFLOW_RUNTIME_CONFIG` or `./runtime_config.json`. TOML is authoritative when its relevant section exists; legacy JSON is fallback-only and must not be edited as if it overrides TOML.

## Production-source proof and safety checks

- The wrapper resolves `$STATE`; flow-memory's `DefaultPathProvider` then resolves `FLOW_MEMORY_DB`, or `$STATE/flow_memory.db`. The live backend is `flow_memory.storage.sql.SqliteBackend`, package 0.1.0.
- Editable source authority: source module `/Volumes/Halobster/Obsidian Edu/留学公司知识库/11-Eduflow Team 多智能体项目/flow-memory/src/flow_memory/storage/sql.py`; source git HEAD `39920700c40c088d77695da5114d970047af4cac`; dirty=true; source SHA-256 `64cf9118d7ce266542ca6abdb387f4dd9f866ded1a6f19bd61284603108879d2`. Package version alone is not revision authority while the editable checkout is dirty.
- Deterministic schema fingerprint `66293588bd5f0525c403f2df97c81447972723046981703a19d3881a2a573701` hashes ordered SQLite object type/name/table/normalized SQL metadata. The live DB has 38 non-internal sqlite_schema objects; this is an object count, not a claim of 38 application tables. `user_version=0` is a missing migration-version risk.
- Key schema metadata: `memory_items` columns `id, layer, scope, kind, status, content, summary, source_ref, evidence_refs, confidence, importance, valid_from, valid_until, created_by, created_at, updated_at, supersedes, revision_of, metadata_json, pinned, source_agent`; indexes `idx_mi_kind, idx_mi_layer, idx_mi_scope, idx_mi_status, sqlite_autoindex_memory_items_1`.
- `active_constraints` columns `id, scope, constraint_level, constraint_type, content, source_ref, evidence_refs, status, enforcement, injection_point, valid_from, valid_until, created_by, created_at, updated_at`; indexes `idx_ac_level, idx_ac_scope, idx_ac_status, sqlite_autoindex_active_constraints_1`.
- `task_capsules` columns `task_id, workflow_id, owner, gate, goal, acceptance, current_status, decisions, blockers, next_action, last_evidence_ref, updated_at`; index `sqlite_autoindex_task_capsules_1`.
- `sensitive_memory_items` columns `id, scope, kind, encrypted_data, nonce, tag, status, created_at, updated_at`; indexes `idx_smi_scope, idx_smi_status, sqlite_autoindex_sensitive_memory_items_1`.
- `src/eduflow/runtime/paths.py` still resolves legacy `$STATE/eduflow_memory.db`, task events, runtime status, scheduler files, and unified config. `src/eduflow/memory/db.py` owns that legacy DB and forces 0600; it is not the active flow-memory authority. An unknown active memory DB location is a deployment failure.
- `src/eduflow/runtime/agent_auth.py` proves the agent credential source and resolution priority without requiring disclosure of secret values. An unknown credential source must block launch/migration rather than be guessed.
- `src/eduflow/feishu/lark.py` creates `$STATE/.tenant_token.json` with mode 0600 and otherwise delegates host authentication to lark-cli. Never print or commit token, key, password, secret, or credential contents while checking these paths.

Before backup, restore, or migration, capture the live process `EDUFLOW_STATE_DIR` and `EDUFLOW_CONFIG_FILE`, resolve both paths, and compare them with the wrapper target. A directory existing on disk is not proof that production reads it.
