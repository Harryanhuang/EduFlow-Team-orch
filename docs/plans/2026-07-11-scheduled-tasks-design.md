# 定时任务（D 编号）设计

**状态：** 已确认设计，待实施。

## 目标

让 user 能创建单次、每日或每周的定时业务规则；每次到期均先交给
manager 确认，再由 manager 编排一个或多个执行 agent。定时规则及其
每次对外执行均使用 `D-<序号>`；既有非定时业务继续使用 `T-<序号>`。

## 设计边界

- V1 仅支持单次、每日、每周；展示时区固定 `Asia/Shanghai`，持久化
  时间使用 UTC。
- 只有 user 可创建、暂停、恢复、取消规则；manager 可决定本轮是否
  执行以及如何派发；worker 只能回报。
- 定时规则、轮次和锁是调度存储的事实来源。memory 只提供上下文和
  审计摘要，不能驱动调度状态变化。
- 不把 D 轮次伪装成现有 T 任务；D 使用专用执行适配层和内部 lane。
- manager 保持派发/审核职责。若生成 workflow 资产需要文件改动，由
  `worker_builder` 执行普通内部维护任务，manager 审核、启用并通知 user。

## 数据模型

### 规则

`scheduled_tasks` 保存 `D-<id>`、任务目标、预期产物、频率、时区、
下一次到期 UTC 时间、状态、默认并发策略、workflow 状态和版本号。
`D-` 计数器独立于现有 task store 的 `T-` 计数器。

### 轮次与 lane

每次计划执行生成一个 `scheduled_occurrence`，其幂等键为：

```text
D-<id>:<scheduled_at_utc>
```

一轮可有一个或多个内部 lane；lane 幂等键追加 lane 序号。对 user 只展示
`D-<id>`，但审计记录必须保留每个 lane 的 agent、依赖、结果和失败原因。

轮次状态至少包括：`detected`、`awaiting_manager`、`awaiting_user`、
`dispatching`、`running`、`completed`、`failed`、`skipped`、`overdue`、
`blocked_by_previous_run`。

## 交互与时间语义

1. user 的自然语言请求先生成草稿。manager 必须补齐目标、产物、时间/频率
   与时区；“下周”“周末”“下午”“隔一段时间”等模糊表达必须追问。
2. user 确认草稿后才持久化规则。暂停、恢复、取消也需回显确认，且确认令牌
   绑定规则版本，防止旧消息改变新规则。
3. 到期后轮次进入 manager 待确认队列，绝不自动派发。manager 选择 agent、
   lane 和串并行关系；成功派发后通知 user “本轮已开始执行”。
4. manager 选择不执行时，user 只能选“跳过本轮”或“取消 D 规则”；无回复
   默认跳过本轮、保留周期规则。
5. 单次任务待 manager 确认时，每 30 分钟提醒 manager；两小时后通知 user；
   不自动取消。
6. 周期任务的一轮尚未确认而下一轮到期时，旧轮标记为 `overdue/skipped`，
   新轮取代它；同一规则最多一个未确认轮次。
7. 服务停机错过的轮次恢复后先询问 user 是否补执行；同意后回到 manager 确认。
8. 失败不无限重试。manager 可重派、跳过或暂停规则。

## 容量、重叠与背压

同一 D 默认最多一轮处于 `dispatching/running`。若下一轮到期时上一轮已经
执行中，新轮进入 `blocked_by_previous_run`，由 manager 决定跳过、排队或明确
允许并行；默认不并行。这与“未确认跨下一轮自动跳过”是两条独立规则。

规则还需保存：最大并发、最大待处理轮次数、连续跳过/失败阈值。超过阈值时
状态转为 `attention_required`，停止产生新的自动待确认轮次，等待 manager
处理。manager 面板应显示队列积压、最长等待、执行中重叠和容量告警。

## workflow 演进

新规则从 `exploration` 开始。manager 在每轮确认时按当前上下文制定协作方案，
并记录 agent 编排、依赖、产物、人工调整和失败原因。

连续至少 5 次稳定完成后，才可产生 workflow 候选。稳定要求：目标和预期产物
一致、关键步骤/角色稳定、结果可审计，且没有未解决的重复失败或重建。

manager 触发候选生成与严格验证；如需写 workflow 资产，交给 `worker_builder`。
manager 审核后直接启用并通知 user。复用 workflow 后，每轮仍必须经过 manager
确认；workflow 仅提供协作编排快照，不自动执行。

出现连续两次重大偏离或失败时，自动从 `workflow_reuse` 降级回 `exploration`：
写入原因、失效快照和 user 可见说明，当前轮不自动重试。除此之外，系统每 30
天或每 10 个成功轮次（先到者）要求 manager 对近期样本做一次 workflow 健康检查，
避免没有报错但已逐步失配的“静默漂移”。

## 运行时与恢复

`task-publish` 每轮除现有发布工作外调用独立、异常隔离的 scheduler tick。定时
扫描使用独立游标、锁与心跳；调度异常不得停止普通任务发布。状态必须先写入，
再通知或派发。进程重启时 reconciliation 负责恢复中间状态、补发缺失通知、处理
错过轮次，且不得重复派发。

health 增加 scheduler 的最近成功 tick、扫描滞后、待 manager 确认、待 user
决定、连续跳过/失败、执行中轮次和积压项指标。

## 记忆与历史保留

为 `D-<id>` 建立任务胶囊，保留规则摘要、稳定模式、workflow 启停和重大失败。
每轮的完整 lane/通知/产物证据留在调度执行历史；日常 tick、提醒和普通等待不写入
memory。执行历史按“近期可检索 + 长期摘要归档”保存：建议 90 天保留完整轮次，
之后保留聚合摘要和可定位的证据引用，具体时长在实施时配置化。

## 验收重点

- D/T 存储、编号、状态机和对外展示互不污染。
- 重复 tick、重启恢复、通知失败、取消与派发竞态均不导致重复执行。
- 执行中重叠、容量阈值和长期无人处理会收敛到可见告警状态。
- 五次稳定运行才形成候选；两次偏离/失败或定期健康检查可退出复用。
- scheduler 故障不会影响既有 task-publish 常规任务发布。
